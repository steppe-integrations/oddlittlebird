Why this document exists

Two halves of one pipeline, discovered independently

You run a four-gate pipeline that keeps published documentation true after it ships. I build engineering systems that emit timestamped, citable artifacts as a byproduct of the work. On the podcast, these turned out to be the same thesis approached from opposite ends. This page is the follow-up — my proposal for how we can structure a collaboration: evidence first, ask second.

The one-line thesis. Machine-readable documentation is a formatting problem. Machine-trustworthy documentation is a process problem — and the process has two halves:

Traceable upstream, verified downstream — documentation agents can trust because it carries evidence at creation and passes tests in production.

Almost nobody owns the seam between the halves: turning an artifact trail into published pages that stay tied to the artifacts they came from, while both keep changing. This dashboard is the operating surface for finding out whether the two of us can.

The proposal, in one breath

  1. A 20–30 minute screen swap — you show the VMs and your Claude Code setup, I show artifacts generating themselves as work happens. Cameras on, no slides.
  2. A 60–90 minute working session — one simulated product change: a coding agent drafts the doc update plus its Doc Detective test from my artifacts, and the PR must survive your four gates in public CI. You drive, co-drive, or watch and heckle.
  3. Co-author the writeup — a first draft exists; it ships once it can cite a real merged PR instead of a promise.

03The contribution — a Documentation Trust API

What the two standards look like as one cohesive implementation. Source of truth: spec/trust-api.yaml (OpenAPI 3.1) · decision record: ADR-0001, which scores the four alternatives this beat. Click an operation to expand it.

Why this shape — the convergence

The two repos already built the same ideas from opposite directions. The spec names them:

commandcenter (my side)showcase-site (your side)Named in the spec as

The one-move repair. Your gap (no provenance model) is filled by my artifact graph; my gap (implicit, timestamp-organized) gets forced into a computable schema by your gates. Neither repair works without the other party.

GET/manifest The only object that must cross an org boundary

A lockfile for provenance: one entry per exportable artifact — id, type, unified status, source commit, ADR references, digest. When engineering and docs live in different orgs, tools, or trust domains, this manifest is the entire interface. Regenerate by cron, webhook, or hand.

200Current manifest, small and diffable.
POST/verifications Gate runs as immutable ledger records

CI is the caller. Four gates — style, spelling, links, accuracy — each write a record per run: actor, outcome, digest, timestamp. Corrections append; nothing edits history.

201Recorded, immutably.
GET/coverage The failure-mode ledger — evidence-linked, not hypothetical

Rows are concrete failure modes; columns say which practice prevents each, with links to real retros and real gate failures. Uncovered rows become ADR proposals — gaps feed the roadmap instead of hiding in a slide deck. Costs a markdown table to start; survives every adoption blocker.

When verification goes stale or a cited ADR is superseded, trust decays past threshold and this webhook files a ticket against the owning artifact in the engineering repo's own planning surface.
